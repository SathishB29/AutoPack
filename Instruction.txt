# GitHub Copilot Instructions — AutoPack

You are working in this repository as the principal engineer building **AutoPack**, an open-source **Automotive Investigation Pack Builder**.

Your job is to build a real product, not a demo, not a thin wrapper, and not a chatbot over raw logs.

---

## 0. Product Mission

Build a **CLI-first deterministic evidence compiler** for automotive validation.

AutoPack ingests noisy automotive artifacts such as:
- DLT logs
- BLF / ASC bus traces
- DBC files for decoding
- optional TCU platform logs (syslog / Linux / QNX-style text logs)
- optional test logs / test artifacts (secondary only)

AutoPack outputs a **portable investigation pack** that:
- humans can inspect
- scripts can process
- CI can archive
- LLMs can reason over later

### Product slogan
**Record once. Investigate anywhere.**

### Core principle
**Deterministic preprocessing first. LLM reasoning later.**

Never build a product that depends on pasting raw 20k-line logs into a model.

---

## 1. Non-Negotiable Design Principles

### 1.1 No thin wrappers
Do not build:
- a raw log upload UI
- a generic “AI log analyzer”
- a prompt wrapper over GPT/Copilot/Codex
- a fake RAG system over raw log lines

If the feature is just “take file, call LLM, print summary”, reject it.

### 1.2 Deterministic first
The core value must exist **without any LLM**.

The repository must still be useful if no model/API is configured.

### 1.3 Evidence > prose
Every conclusion must point back to structured evidence:
- timestamps
- source file
- event id
- decoded signal
- anomaly window
- correlation rule

### 1.4 Token efficiency is a systems problem
Primary token reduction must come from:
- parsing
- normalization
- deduplication
- burst collapsing
- anomaly extraction
- timeline slicing
- evidence selection

Not from “just use smaller prompts.”

### 1.5 CLI-first
The CLI is the product center.

Everything else is secondary.

### 1.6 Open-source quality
This repo must be:
- understandable
- well-tested
- modular
- documented
- benchmarked
- usable by engineers without LLMs

## 2. Product Scope by Phase

# Phase 1 — Automotive Only (Primary Scope)
Focus only on **TCU / ECU validation incident triage**.

### Inputs for Phase 1
Required:
- DLT log input
- BLF or ASC input
- DBC input for decoding
Optional:
- syslog / Linux / QNX text log
- test logs / test artifacts (secondary)

### Phase 1 Outputs
Create an **investigation pack** containing:
- manifest.json
- timeline.jsonl
- anomalies.jsonl
- correlations.jsonl
- summary.md
- source-derived parsed artifacts
- optional evidence windows

### Phase 1 must NOT include
- dashboards
- web UI
- MCP-first workflow
- embeddings-first retrieval
- historical vector search
- multi-vehicle everything
- automatic fix generation
- “AI root cause magic”

## 6. Core Domain Model

Implement a canonical normalized event schema.

Each event should support fields like:
- event_id
- timestamp
- source_type
- source_file
- ecu
- app_id
- context_id
- bus_channel
- frame_id
- signal_name
- severity
- category
- message
- attributes (free-form structured map)
- tags
- correlation_keys

Never keep only raw text if structure can be extracted.

---

## 7. Pack Format

The pack is the main product artifact.

Implement a pack format like:

incident-pack/
  SESSION.md
  DATA_FORMAT.md
  INVESTIGATION_GUIDE.md
  manifest.json
  timeline.jsonl
  anomalies.jsonl
  correlations.jsonl
  summary.md
  sources/
    dlt.parsed.jsonl
    bus.frames.parquet
    bus.signals.parquet
    syslog.parsed.jsonl
    test_failures.jsonl
  evidence/
    top_patterns.json
    timeline_windows.json
    root_cause_candidates.json

### Rules
- Keep machine-readable and human-readable outputs together
- Use JSONL for event streams
- Use Parquet where large tabular/bus data is useful
- Do not hide information in opaque binary blobs
- Include source hashes and parse stats in manifest.json

## 8. Phase 1 Feature Requirements

Implement at minimum:

Implement deterministic pipelines for:

- anomaly extraction

### 8.4 Required summaries

- candidate trigger windows

## 9. Test Logs Policy

Test logs are supported as **secondary evidence** only.

They are not the product center in Phase 1.

Use test logs to capture:
- which test failed
- failure step
- failure message
- timestamps
- retry/flaky information
- artifact references

Never treat test report output alone as root cause.

Implement a normalized test evidence format:
- test_id
- framework
- status
- duration
- failure_kind
- message
- stack_summary
- artifact_refs
- linked_windows

## 10. LLM / Retrieval Policy

### 10.1 Phase 1 default
No LLM required for core functionality.

No embeddings required.
No reranker required.
No MoE-specific architecture.

### 10.2 LLM usage policy
Only use an LLM after deterministic pack generation, for:
- explanation
- Q&A over pack
- evidence-grounded root-cause hypotheses

Never use an LLM for:
- raw parsing
- DBC decoding
- timestamp alignment
- repeated pattern grouping
- burst detection
- simple anomaly classification

### 10.3 Retrieval policy for later phases
If retrieval is added later:
- index only normalized evidence, summaries, and historical packs
- do NOT vectorize raw noisy log lines by default
- use rerankers only after deterministic filtering + first-stage retrieval

## 12. Coding Standards

### 12.1 General
- Prefer explicit code over clever code
- Favor pure functions for transformation stages
- Keep parsing separate from normalization
- Keep normalization separate from explanation
- Keep optional AI integrations behind clear interfaces

### 12.2 Python specifics
- Use type hints everywhere
- Use dataclasses or pydantic models where appropriate
- Use pathlib instead of string paths
- Keep dependencies minimal and justified
- Use logging, not print, for library code
- Provide clear CLI error messages

### 12.3 Performance
- Support large files without exploding memory where possible
- Prefer streaming / iterators for raw event reading when practical
- Use Parquet for large decoded tables when useful

## 15. Acceptance Criteria for Phase 1

Phase 1 is successful only if all of the following are true:

- A user can run `autopack build` on a real or fixture incident
- The command outputs a valid investigation pack
- The pack is readable without AI
- The pack meaningfully reduces raw evidence volume
